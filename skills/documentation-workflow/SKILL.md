# Documentation Workflow

A structured workflow for writing and updating documentation for every feature that needs it. Triggered as Stage 4 of the pipeline. Assesses documentation needs based on change size before writing.

---

## Pipeline Position

```
spec-workflow > testing-workflow > code-review + security-review > [documentation-workflow] > git-workflow > deployment-workflow
```

Triggered by: spec-workflow (after code-review and security-review pass), deployment-workflow (pre-deployment doc gate)
Signals back to: spec-workflow

---

## Pipeline State

On start: read `tasks/pipeline-state-[feature-name].md` to understand change size, pipeline path, and review results.
On complete: update the state file with documentation result and timestamp.

---

## ECC Agent Delegation

This skill delegates to ECC agents for automated documentation generation. Delegation is optional and enhances -- never replaces -- manual documentation review.

| Step | ECC Agent | Purpose | When to Use |
|------|-----------|---------|-------------|
| Step 1 (Audit) | `doc-updater` | Codemap generation, automated doc extraction from code, identifying documentation gaps | Standard+ changes with significant new code |

Agents NOT used here (and why):
- `architect`: Architecture docs are written based on the spec and implementation knowledge, not generated by the architect agent
- `code-reviewer`: Documentation quality is this skill's responsibility, not code-review's

---

## How to Start

Detect context:
- Triggered from spec-workflow > assess docs needed, then document the new feature
- Triggered from deployment-workflow > verify all documentation is current before release
- Standalone > ask what needs to be documented

Load `tasks/spec-[feature-name].md` and project CLAUDE.md before starting.

---

## Step 0: Documentation Needs Assessment

Not every change needs full documentation. Assess based on change size and type:

```
Trivial changes (typo, config, formatting):
  Documentation needed: None
  Signal: "Documentation not required for [feature]. Trivial change. Proceed to git-workflow."

Small changes (bug fix, minor feature):
  Documentation needed: Changelog entry only (if user-visible behaviour changed)
  Skip: README, API docs, architecture docs, inline comments review
  If no user-visible change: signal "Documentation not required"

Standard changes (new feature, new endpoint, new integration):
  Documentation needed: Full assessment (Steps 1-6)
  All documentation types evaluated

Critical changes (security fix, auth change, infrastructure):
  Documentation needed: Full assessment + explicit security documentation
  Update any security-related docs, runbooks, or incident procedures
```

If documentation is not required: update pipeline state file and signal back immediately. Do not proceed to Step 1.

If documentation is required: proceed to Step 1.

---

## Step 1: Documentation Audit

**ECC delegation:** For Standard+ changes with significant new code, spawn the `doc-updater` agent as a subagent with the changed files. It will generate a codemap of the new/modified components and extract documentation from code (docstrings, type hints, function signatures). Use its output to identify documentation gaps more accurately. The agent generates raw material -- you still decide what gets documented and how.

Assess what exists and what needs updating:
- README -- does it exist? Is it up to date?
- API documentation -- does it reflect current endpoints?
- CHANGELOG -- does it exist? Is it current?
- Inline code comments -- are complex sections explained?
- Architecture docs -- do they reflect any new components?
- Setup and onboarding -- are instructions still accurate?

List what needs to be created or updated. Present to user. Get confirmation before writing.

---

## Step 2: README Update

Update README if the feature changes how the project is used, set up, or configured:
- Project description
- Tech stack overview
- Prerequisites and dependencies
- Setup and installation -- step by step, tested
- How to run locally
- How to run tests
- Environment variables required (names only, no values)
- Deployment instructions or link to deployment docs
- Known limitations or gotchas

Rules:
- README must be accurate -- wrong documentation is worse than no documentation
- Test every setup step before documenting it
- Never include real secrets even as examples

Skip if the feature does not change setup, config, or usage.

---

## Step 3: API Documentation

For every API endpoint added or changed by this feature:
- Endpoint path and HTTP method
- Description of what it does
- Authentication required (yes/no, type)
- Request parameters (path, query, body) -- name, type, required/optional
- Request body schema with example
- Response schema with example for each status code
- Error responses and what they mean

Follow the project's existing API doc format. If none exists, use Markdown.

Skip if no API changes were made.

---

## Step 4: Changelog Update

Add a CHANGELOG entry for every user-visible behaviour change:

```
## [version or date]

### Added
- [new feature or capability]

### Changed
- [behaviour that changed]

### Fixed
- [bug that was fixed]

### Removed
- [functionality removed]

### Security
- [security fix or improvement]
```

Rules:
- Every user-visible change gets an entry
- Internal refactors with no behaviour change do not need a changelog entry
- Write for humans -- plain English, not technical jargon
- Changelog must be committed alongside the code change

Skip if the change is purely internal with no user-visible impact.

---

## Step 5: Inline Code Documentation

Review code for missing or outdated inline comments:
- Every non-obvious function or method needs a comment explaining what and why
- Complex algorithms or business logic explained inline
- Workarounds documented with reason and ideally a link to the proper fix
- Outdated comments removed
- Do not comment obvious code -- comments explain intent, not mechanics

---

## Step 6: Architecture Documentation

If the feature changed the system structure:
- New component added > document it
- Data flow changed > update data flow description
- New integration added > document purpose, behaviour, and failure modes
- Infrastructure changed > update infrastructure docs

Save or update to `docs/architecture.md` at the project root.

Skip if no architectural changes were made.

---

## Step 7: Complete

- Confirm all documentation is accurate by cross-referencing with spec and implementation
- Update `tasks/todo.md` -- mark documentation complete
- Update `tasks/pipeline-state-[feature-name].md`: "documentation-workflow: complete [timestamp]"
- Update `tasks/lessons.md` with documentation patterns discovered

Signal back to spec-workflow:
- Success: "Documentation complete for [feature]. Pipeline Stage 4 passed. Proceed to git-workflow."
- Not required: "Documentation not required for [feature]. Trivial/internal change. Proceed to git-workflow."
- Blocked: "Documentation blocked for [feature]. Reason: [X]. Returning to spec-workflow."

If triggered as deployment gate:
- Success: "Documentation deployment gate passed. Returning to deployment-workflow."
- Blocked: "Documentation deployment gate incomplete. Reason: [X]. Returning to deployment-workflow."

---

## Failure Handling

**README setup steps do not work**
- Test every step from scratch before documenting
- Fix the steps that fail
- If a step works locally but may not elsewhere: document the prerequisite explicitly

**API documentation out of sync with implementation**
- The implementation is the source of truth -- update docs to match
- Flag any endpoints documented but not existing > remove them
- Flag any endpoints existing but not documented > add them

**Changelog has gaps from previous features**
- Backfill missing entries from git history and spec files
- Going forward: treat missing changelog entry as a documentation blocker

**Documentation is outdated and unclear what is current**
- Add "Last verified" date to README and architecture docs
- Make documentation review part of every deployment checklist
- Outdated docs are treated as a bug -- fix before deploying

**No documentation exists at all (existing project)**
- Start with README and setup instructions
- Then API docs for public endpoints
- Then architecture overview
- Document incrementally -- a little now is better than nothing

---

## Rules

- Documentation needs are assessed based on change size -- not every change needs full docs
- Trivial changes need no documentation
- Small changes need at most a changelog entry
- Standard and Critical changes get full documentation assessment
- README setup steps must be tested before documenting
- Every API change must be reflected in API docs
- Every user-visible change must have a CHANGELOG entry
- Never include real secrets in documentation
- Architecture docs must be updated when system structure changes
- All documentation changes committed via git-workflow as part of the same PR
- Always update `tasks/pipeline-state-[feature-name].md` with results
- Always signal back explicitly -- complete, not required, or blocked with reason
